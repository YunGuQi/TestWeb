### 阶段一：创意构思与机制设计 (Brainstorming)
1. **【强制前置动作】** 收到指令后，**必须**启动 `grill-me`、`brainstorming` 或 `interview-me` 技能。绝对禁止在未进行多轮“一问一答”（One-by-One）需求对齐前，直接输出方案、代码或 `implementation_plan.md`。请先抛出第一个引导性问题！
2. **题目设计**：根据具体的测试主题和受众耐心程度，分析并决定最合适的题目数量，设计极具代入感的沉浸式选择题。
3. **维度与算法**：根据测试的专业深度，灵活规划最合适的评价维度（具体分析再决定多少维度合适），并构建对应的加减分或权重映射逻辑。
4. **结果池 (Result Pool) 构建**：根据算法组合，设计合理数量的测试结果种类。每个结果需包含：类型名称、专属 Tag、一句极具共鸣的“扎心”或治愈文案。

### 阶段二：高级视觉与 UI 设计 (Design Taste)
1. **【强制动作】** **必须**首先使用 `view_file` 工具阅读 `C:\Users\GU\.gemini\config\skills\taste-skill\SKILL.md` (或相应的风格技能)。绝对不能凭空捏造审美，必须输出 Design Read 并在代码中践行其高阶设计规范。
2. **【防雷同机制】** 在生成 UI 前，必须确保当前的设计与之前的测试（如“性格城市匹配测试”、“情绪内耗测试”）在视觉风格、排版、组件库选用上形成**极其显著的差异化**。禁止复用相似的进度条、选项卡和结果页模板！
3. **视觉要求**：
   - 使用强排版对比（巨大的标题、微小的修饰性 Tag）。
   - 添加背景噪点（Noise Filter）或特殊纹理来增加质感。
   - 包含平滑的微动画（Micro-interactions）和过度效果。
   - 结果页必须像一个真实的、具有反差感的物理凭证或特殊文档。**具体载体（如车票、档案、调令等）需根据测试内容的调性灵活设计，绝不局限于某一种固定形式。**

### 阶段三：前端架构与核心开发 (Code Implementation)
1. **单文件与目录规范**：不要搭建 React/Vue 脚手架，请使用纯粹的单文件 `index.html`。**必须将新的测试项目文件夹创建在 `common/` 目录下（如 `common/xxx-test/index.html`）**，绝对不要直接创建在根目录！
2. **组件流转**：包含“首页 (Landing)” -> “答题页 (Quiz)” -> “结果生成页 (Result)”。
3. **海报生成**：务必引入 `html2canvas.min.js`，实现长按/点击按钮将结果页 DOM 转化为高清图片，以便用户分享。
4. **历史记录展示（强制标准）**：利用浏览器的 `localStorage` 缓存用户的测算结果。**在首页 (Landing Page) 必须提供「历史记录 / 历史测算」的查看入口按钮**，点击后可直接跳转到结果页回显上一次的测算数据。
5. **次要操作弱化与按钮排序（UI标准）**：
   - 结果页底部的操作按钮**必须严格遵循以下垂直排序**：
     1. 【最核心行动点】“保存车票/海报分享”（高对比度主按钮）
     2. 【跨界引流】“探索其他测试”（通过 test-matrix 动态注入的**强视觉按钮，样式与上方主按钮保持统一**，保证引流转化率）
     3. 【次要操作】“重新测试 / 重新启程”（最底部的弱化按钮，**仅保留重新测试的文案，绝对不要带上“返回首页”或“历史记录”等字眼**）
   - 在首页或结果页除了醒目的主行动按钮（或引流按钮）外，极次要按钮（如重新测试）**必须采用弱化设计（例如纯文字+下划线 `underline` 样式，文字较小且颜色较灰）**，绝对不要使用与主按钮抢视觉焦点的带边框或大色块按钮。
6. **防误触与“上一题”机制（核心体验）**：必须在答题页 (Quiz Page) 提供“上一题”撤回功能。底层逻辑必须采用“数组进栈/出栈 + 动态重新算分”机制 (`selectedHistory`)，保证分数撤回绝对精准。按钮应当设计为低调的文字+下划线样式，仅从第二题开始显示。
7. **Hash 路由防退出拦截（防流失机制）**：必须在单页应用中引入 Hash 虚拟路由拦截（如 `#landing`, `#quiz`, `#result`）。通过监听 `window.onpopstate` 或 `hashchange` 事件，**确保用户在手机端左滑返回或点击系统后退按钮时，不会直接退出测试网页，而是被无缝拦截为执行“上一题”或平滑退回首页。**

### 阶段四：测试矩阵引流与置顶逻辑 (Cross-Promotion)
1. 在页面的底部结构中，预留 `<div id="test-matrix-container" class="w-full"></div>` 容器。
2. 确保在生成海报的主要按钮下方，引入公共的测试矩阵模块（`../common/test-matrix.js`）。并在结果页显示时，必须调用 `window.initTestMatrix({containerId: 'test-matrix-container', currentId: '你的测试ID', theme: 'light'})` 方法初始化引流组件。首页也需要引入“探索其他测试”按钮。
3. 确保此测试在矩阵列表中被正确识别，并**自动置顶在引流列表的第 01 位**，带上醒目的 `[当前]` 标签。同时，测试开发完成后，AI 必须主动**调用内置浏览器（browser_subagent）完整跑通一遍测试流程**，并在结果页截图保存为 `preview.jpg` 放置于当前测试项目目录下，然后在 `test-matrix.js` 的数据源中配置 `previewImg` 字段以供预览调用。
4. **纪律警告**：后续如需调整 HTML 结构，请务必直接使用编辑器工具（如 `replace_file_content`）修改，**绝对禁止使用 Python 脚本等间接手段篡改前端文件**。

### 阶段五：小红书发帖与传播物料策划 (Social Operation)
1. 前端代码开发并验证通过后，请自动调用 `xhs-social-content` 等相关技能。
2. 为刚刚生成的测试撰写 3 个极具爆款潜质的**小红书笔记标题/钩子 (Hook)**。
3. 提供一篇完整的小红书正文排版文案，带好适当的 Emoji 和话题 Tag（#心理测试 #周末去哪儿 #人格测试 等）。
4. （可选）为向微信私域或主页店铺引流提供话术建议。
