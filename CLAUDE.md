# 小红书虚拟测试运营 · 核心工作流规范

## 1. 核心目标
寻找、抓取并深度拆解小红书同赛道（心理测试/人格测试/趣味测试）的爆款帖子与对标账号，将其方法论沉淀为可直接模仿的知识资产。

## 2. 抓取与分析工作流
当用户要求寻找爆款并进行对标分析时，必须严格遵守以下防封控的安全闭环步骤：

1. **灰豚数据初筛（纯 UI 操作防封号）**：**绝对禁止**使用 Python 脚本直接抓取灰豚数据接口。必须使用内置浏览器智能体（`browser_subagent`）人工访问灰豚后台，定位目标爆款测试产品，并从网页上直接提取其**关联笔记数**和**关联达人数**。
2. **红书原贴定位**：拿到灰豚上的爆款名称或达人名后，切换使用 `xhs-openclaw` 技能（通过 MCP 调用），在小红书内定向精准搜索对应的关联笔记和博主主页。
3. **账号商业判定**：利用 `xhs-openclaw` 抓取信息判断该博主是否具备商业闭环能力（满足任意一条即可）：
   - 主页简介中包含"测试链接"、"主页链接"、"店铺"、"私信获取"等引导语。
   - 笔记或评论区中出现明确的引流或变现话术。
   - 粉丝量在 1w-20w 之间，且内容高度垂直于测试赛道。
   - 笔记风格统一、封面模板化（说明有批量产出能力和商业规划）。
4. **深挖与爆款拆解**：利用 `xhs-openclaw` 获取该爆款笔记的详情（原图、完整文案、热门评论区的转化套路），深度分析其爆火的底层逻辑（情绪痛点、好奇心钩子等）。
5. **输出报告**：将所有的抓取数据和拆解分析，整理成标准的 Markdown 文档。

## 3. 报告输出规范 (红线)
- **保存路径**：所有的分析报告必须保存在根目录的 `对标账号/` 文件夹中（如果该文件夹不存在则创建）。
- **文件命名**：使用爆款测试的核心主题命名。例如：`森林潜意识测试.md`、`MBTI椅子测试.md`。
- **文档结构要求**：文档中**必须包含**以下模块：
  - **博主名及变现方式**：该爆款帖子的创作者名称，标注其如何变现（主页链接/店铺/私域/挂车等）。
  - **帖子链接**：爆款帖子的真实 URL 链接。
  - **帖子截图**：嵌入抓取时生成的帖子全貌截图。
  - **主图分析**：首图的视觉设计、文案、以及画面描述（有原图更佳）。
  - **帖子标题**：完整的标题文案。
  - **帖子内容**：完整的正文文案。
  - **评论区生态**：热门评论摘录，以及互动套路分析。
  - **爆款原因分析**：深度拆解它为什么能火（痛点、好奇心、视觉表现等）。
  - **如何模仿与导流（落地指南）**：我们将如何复刻它的首图、文案和评论区诱饵，以及它的流量转化路径。

## 4. 测试页面开发规范
- **直接修改 HTML 文件**：修改测试页面（如 `emotional-friction-test` 下的单文件 HTML）时，**必须直接使用编辑器工具修改 HTML/CSS/JS 代码**，**绝对禁止**编写 Python 脚本（如 `rewrite.py`、`patch.py`）来间接修改 HTML 文件。
- **单文件原则**：每个测试产品保持 HTML + CSS + JS 全部内联在一个 `.html` 文件中，便于直接部署和分享。

## 5. 交互与代码操作规范
- **称呼规范**：在与用户对话回复时，必须称呼用户为“大哥”。
- **禁止自行 Git 回滚**：不要自己用 git 回滚代码（如使用 `git checkout` 等命令），以避免已修改或未提交的内容被意外覆盖。
- **字符编码规范**：在修改或处理文件内容时，务必注意 UTF-8 的编码问题，严禁再次出现因编码设置不当导致的中文乱码。**特别警告：绝对禁止在没有显式指定 `-Encoding utf8` 的情况下，使用 Windows PowerShell（如 `Get-Content` 或 `Set-Content`）直接处理包含中文的代码文件，这会将其错误编码并直接导致乱码毁坏文件！优先使用内部的原生文件修改工具（如 `replace_file_content`）。**
